# 个人生活助手 — 需求分析文档

**文档类型**：需求分析 (PRD)  
**创建日期**：2026-03-03  
**状态**：Active (Refactoring from Smart Workspace)  
**目标**：将原有的“智能工作台”重构为以 **AI 能力为核心**的“个人生活助手”。通过自然语言交互，集成日程、邮件及生活服务（天气、票务、外卖等）。

---

## 1. 现有功能总结 (Legacy Features)

基于原《智能工作台助手》项目，目前已具备以下核心能力：
- **用户账密体系**：注册、登录、登出、Session 保持、访客模式（MySQL 持久化）。
- **日程/待办管理**：待办项与日程的 CRUD，支持今日/本周视图。
- **QQ 邮箱基础集成**：邮件列表拉取、邮件发送（后端授权管理）。（未实现，暂不处理）
- **基础 AI 对话能力**：接入阿里云百炼（Qwen），支持 SSE 流式输出，具备基础的自然语言理解能力。

---

## 2. 重构计划：个人生活助手定位

### 2.1 首页重构 (Homepage Refactor)
- **目标**：删除原有的“会话中心”独立页面，将首页改为“个人生活助手”聚合页。
- **变更点**：
  - **删除冗余路由**：移除 `/chat` 或 `/conversation` 等独立页面路由及相关冗余代码。
  - **AI 助手常驻化**：AI 助手不再作为一个页面，而是以**侧边栏 (Sidebar)** 或 **浮动对话框** 形式存在于所有页面，随时待命。
  - **生活化看板**：首页突出展示“今日天气”、“今日提醒”、“生活快捷入口”等信息。

### 2.2 冗余代码清理
- 清理原 `Conversation` 页面相关的业务逻辑、样式文件及 API 接口。
- 优化侧边栏导航，仅保留：`首页`、`待办事项`、`日程管理`、`邮件管理`。

## 3. 新增 AI 生活化功能 (P1-P2)

### 3.1 AI 查天气 (Weather Service)
- **描述**：用户在 AI 助手输入“明天上海天气如何？”，助手自动获取并返回天气数据。
- **实现方案**：AI 助手识别天气查询意图 -> 后端/MCP 调用第三方天气 API (如和风天气) -> 结构化返回给用户。

### 3.2 AI 查车票 (Travel/Ticket Service)
- **描述**：用户通过自然语言查询火车/高铁票，如“帮我查一下明天下午北京到上海的高铁”。
- **实现方案**：AI 助手提取城市、时间等参数 -> 调用票务 API 或 MCP 工具查询 -> 展示班次列表。

### 3.3 AI 发 QQ 邮件 (Smart Email)
- **描述**：不再需要手动填写复杂的邮件表单，通过自然语言发起。如：“给张三发封邮件，主题是下周计划，内容是...”
- **实现方案**：
  - AI 助手解析收件人、主题、正文。
  - **MCP/工具调用**：助手调用预定义的 `send_qq_email` 工具。
  - 调用现有后端 QQ 邮箱 SMTP 接口完成发送。

### 3.4 AI 点外卖 (Takeout Service - 最后实现)
- **描述**：AI 助手辅助搜索外卖、推荐店铺。
- **实现方案**：作为进阶功能，主要通过 AI 识别菜品需求，并引导至三方外卖平台或展示推荐列表。

## 5. 技术选型变更（确定版）
- **后端架构**：采用 **Node.js 后端直连 MCP Server**（不引入 Dify 作为必需依赖）。
- **AI 助手增强**：使用 **Function Calling + MCP Tool Adapter**，由模型按需触发工具调用。
- **MCP 工具来源**：支持接入「本地安装」或「可下载部署」的 MCP Server（含基础工具、12306、麦当劳）。
- **响应式**：强化移动端体验，生活助手需适配手机端快速操作。

## 6. MCP 接入需求（新增）

### 6.1 目标与范围
- 在不破坏现有登录、待办、日程、基础聊天能力的前提下，为 AI 助手增加可执行工具能力。
- 当前纳入范围：
  - **基础 MCP 工具**：时间、网页搜索、天气（可选）、文件读取（只读模式）。
  - **12306 MCP**：车次查询、余票查询（仅查询，不下单）。
  - **麦当劳 MCP**：门店查询、菜单浏览、活动信息查询（仅查询，不下单）。
- 不在本阶段范围：
  - 自动支付、自动下单、自动登录第三方账号。
  - 未经确认的高风险写操作。

### 6.2 业务流程（统一）
1. 用户输入自然语言请求（如“明天下午上海到杭州高铁有哪些？”）。
2. 后端将消息和工具列表发给千问模型。
3. 模型决定是否触发工具调用（tool call）。
4. 后端路由到对应 MCP Server 并执行。
5. 后端把工具返回结果回灌给模型。
6. 模型生成最终自然语言答复并返回前端。

### 6.3 功能需求（Functional Requirements）

| 编号 | 名称 | 说明 |
| :--- | :--- | :--- |
| FR-MCP-01 | MCP 工具注册 | 后端支持注册多个 MCP Server 及其工具元信息（名称、参数 schema、可用状态）。 |
| FR-MCP-02 | 工具路由执行 | 根据 `tool_name` 路由到对应 MCP Server 并执行调用，支持超时与重试。 |
| FR-MCP-03 | 基础工具集成 | 至少接入 2 个基础 MCP 工具（推荐：`web_search`、`time_now`）。 |
| FR-MCP-04 | 12306 查询能力 | 支持车次/余票查询，输出结构化列表（车次、发车/到达、耗时、余票状态）。 |
| FR-MCP-05 | 麦当劳查询能力 | 支持门店、菜单、活动信息查询，输出结构化结果。 |
| FR-MCP-06 | 失败回退 | 任一工具调用失败时，系统自动回退为纯模型回答，不影响会话继续。 |
| FR-MCP-07 | 安全确认 | 涉及写操作的工具（未来扩展）必须二次确认后再执行。 |
| FR-MCP-08 | 会话可追踪 | 保存每次工具调用日志：会话ID、工具名、参数摘要、耗时、状态、错误信息。 |

### 6.4 非功能需求（NFR）
- **NFR-MCP-01 可用性**：单个 MCP Server 故障不应导致聊天接口整体不可用。
- **NFR-MCP-02 性能**：单次工具调用建议超时 8~15 秒；超时后优雅降级。
- **NFR-MCP-03 安全性**：
  - 第三方 MCP 凭据必须通过环境变量注入，不写死在代码仓库。
  - 限制可调用工具白名单，禁止模型调用未注册工具。
- **NFR-MCP-04 可观测性**：可按工具维度查看成功率、失败率、平均耗时。
- **NFR-MCP-05 可扩展性**：新增 MCP Server 时不应影响既有工具调用逻辑。

### 6.5 接口与配置需求
- 后端需提供统一工具执行抽象（示例）：`executeTool(toolName, args, context)`。
- MCP Server 配置项（示例）：
  - `MCP_ENABLED=true|false`
  - `MCP_TOOL_TIMEOUT_MS=12000`
  - `MCP_SERVERS_JSON=[...]`（包含 server 名称、传输方式、地址/命令）
- 生产环境要求：
  - 默认关闭高风险工具。
  - 默认开启工具调用日志（脱敏）。

### 6.6 验收标准（UAT）
- **AC-01**：当用户询问车票时，助手能正确调用 12306 MCP 并返回候选车次。
- **AC-02**：当用户询问“附近麦当劳/套餐”时，助手能正确调用麦当劳 MCP 并返回可读信息。
- **AC-03**：MCP 故障时，聊天仍可用，用户能收到明确降级提示。
- **AC-04**：日志可回溯一次完整调用链（模型决策 -> MCP 调用 -> 回答生成）。
- **AC-05**：现有功能（登录、待办、日程、普通聊天）无回归。

## 7. 迭代计划（MCP 专项）

### 7.1 P1（最小闭环）
- 接入 2 个基础 MCP 工具（`web_search`、`time_now`）。
- 打通后端工具路由和失败回退。
- 完成日志埋点。

### 7.2 P2（生活场景增强）
- 接入 12306 MCP（查询类能力）。
- 接入麦当劳 MCP（查询类能力）。
- 优化中文意图识别与参数补全（出发地、日期、时段等）。

### 7.3 P3（稳定性与体验）
- 建立工具健康检查与熔断策略。
- 增加工具调用结果卡片化展示（前端）。
- 完善观测面板与告警（按工具维度）。

---

## 8. 风险与约束
- 第三方 MCP Server 质量参差不齐，存在协议兼容与稳定性风险。
- 12306、麦当劳类服务可能涉及接口变更、风控限制或法律合规要求，需仅做“信息查询”。
- 外部服务可用性受网络影响，需预设超时与降级策略。

---

## 9. 术语说明
- **MCP**：Model Context Protocol，模型调用外部工具/资源的标准协议。
- **Tool Calling**：模型输出工具调用意图，后端执行后再回灌模型。
- **降级**：工具不可用时，改为纯模型回答并提示限制。

---

## 10. 结论
当前方案定为：**Node 后端直连 MCP**。  
按“基础 MCP -> 12306 -> 麦当劳”逐步接入，先查后写，先稳后广，确保新增能力不影响原有核心功能。
