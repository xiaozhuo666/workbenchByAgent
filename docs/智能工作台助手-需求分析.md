# 智能工作台助手 — 需求分析文档

**文档类型**：需求分析 (PRD / 需求规格)  
**创建日期**：2025-03-01  
**状态**：Draft  
**目标**：实现一个集成邮件（QQ 邮箱）、日程/待办、会话功能的个人工作台应用，便于后续按模块分步实现。

---

## 1. 项目概述

### 1.1 产品定位

- **名称**：智能工作台助手  
- **定位**：个人效率工作台 Web 应用，集成常用办公能力，可作为简历项目展示对前端、集成与（可选）AI 助手的理解。  
- **核心能力**：  
  - 邮件：使用 QQ 邮箱接收、发送邮件  
  - 日程与待办：创建、查看、管理日程和待办事项  
  - 会话：与智能助手对话（可与 Dify 等后端集成，实现问答、总结、提醒等）

### 1.2 目标用户与使用场景

- **主要用户**：本人（及可选的面试官演示）。  
- **典型场景**：  
  - 在一个页面内查看今日待办与近期日程，并处理邮件与助手对话。  
  - 通过自然语言让助手帮忙“查今天有什么待办”“总结未读邮件”等（若接入 AI）。

### 1.3 范围与边界

- **在范围内**：  
  - QQ 邮箱的收件/发件（需授权与 API 或协议支持）。  
  - 日程/待办的增删改查与基本展示（日/周视图可选）。  
  - 会话界面与消息展示；可选接入 Dify 等大模型 API 实现智能回复。  
- **暂不在范围**：  
  - 多租户、多用户账号体系（可先单用户/本地或简单登录）。  
  - 企业邮箱、其他邮箱厂商（首版仅 QQ 邮箱）。  
  - 移动端原生 App（以 Web 为主，响应式可选）。

## 2. 用户故事与验收场景

### User Story 1 - 邮件收发（QQ 邮箱）(Priority: P1)

作为用户，我希望能用工作台**查看 QQ 邮箱收件列表**并**发送邮件**，这样就不必单独打开 QQ 邮箱网页。

**Why this priority**：邮件是工作台的核心能力之一，无邮件则“工作台”价值不足。

**Independent Test**：配置 QQ 邮箱授权后，能在应用内看到最近 N 封邮件列表，并能成功发送一封测试邮件。

**Acceptance Scenarios**:

1. **Given** 已配置 QQ 邮箱授权，**When** 打开邮件模块或刷新列表，**Then** 展示收件箱列表（发件人、主题、时间等），支持分页或“加载更多”。  
2. **Given** 在邮件模块，**When** 点击“写邮件”并填写收件人、主题、正文后发送，**Then** 发送成功并给出提示，且可在“已发送”或收件箱中验证（视 API 能力）。  
3. **Given** 未配置或授权过期，**When** 进入邮件模块，**Then** 展示引导配置/重新授权的入口与说明。

---

### User Story 2 - 日程与待办管理 (Priority: P1)

作为用户，我希望能**创建、查看、编辑、完成/删除**日程和待办事项，并在工作台首页或日程页看到今日/本周视图。

**Why this priority**：日程与待办是个人工作台的核心，与邮件、会话并列。

**Independent Test**：能创建一条待办并标记完成，能创建一条带日期的日程并在列表中看到。

**Acceptance Scenarios**:

1. **Given** 在日程/待办模块，**When** 新增一条待办（标题必填，截止时间可选），**Then** 列表中展示该待办，并支持“标记完成”“删除”。  
2. **Given** 在日程/待办模块，**When** 新增一条日程（标题、日期、可选时间），**Then** 在列表或日历视图中展示，并支持编辑、删除。  
3. **Given** 已有若干待办与日程，**When** 打开“今日”或“本周”视图，**Then** 按时间或日期聚合展示，便于快速浏览。  
4. **Given** 某条待办已标记完成，**When** 查看列表，**Then** 可区分已完成与未完成（样式或分区），并可选择隐藏已完成。

---

### User Story 3 - 会话功能 (Priority: P2)

作为用户，我希望能与**智能助手对话**，发送消息并看到回复；可选地，助手能根据我的日程/待办或邮件摘要回答问题（依赖后端能力）。

**Why this priority**：会话体现“智能”与集成能力，但首版可先做界面与基础对话，再接 Dify。

**Independent Test**：在会话界面输入一条消息并发送，能收到一条回复（可为固定回复或 Dify 返回）。

**Acceptance Scenarios**:

1. **Given** 在会话模块，**When** 输入文本并发送，**Then** 消息出现在对话列表，并展示助手的回复（流式或一次性均可）。  
2. **Given** 已接入 Dify（或类似 API），**When** 用户问“今天有什么待办？”，**Then** 助手能基于当前用户的待办数据回答（需后端或前端把待办数据传给 API）。  
3. **Given** 网络异常或 API 失败，**When** 用户发送消息，**Then** 界面给出错误提示，且用户可重试。

---

### Edge Cases

- **邮件**：授权过期、网络超时、无邮件时列表为空、发件失败（如收件人格式错误）时的提示与重试。  
- **日程/待办**：无数据时的空状态；跨天/跨时区显示；大量数据时的分页或虚拟滚动。  
- **会话**：长对话的上下文长度限制；敏感信息不落库或脱敏（若持久化）；未接入 AI 时的占位回复说明。

## 4. 关键实体与数据（概念层）

- **邮件 (Email)**：对应一封邮件。属性：发件人、收件人、主题、正文、日期时间、是否已读、邮件 ID 等。来源：QQ 邮箱 API。  
- **待办 (Todo)**：属性：标题、是否完成、截止日期/时间、创建时间、唯一 ID。可仅前端存储或同步到后端。  
- **日程 (Event/Schedule)**：属性：标题、日期、开始/结束时间、备注、唯一 ID。可仅前端存储或同步到后端。  
- **会话消息 (ChatMessage)**：属性：角色（user/assistant）、内容、时间戳、可选会话 ID。用于展示与可选持久化。  
- **用户/授权 (User / MailAccount)**：当前用户身份与 QQ 邮箱授权信息（如 token、刷新方式），用于邮件拉取与发送。

---

## 5. 非功能需求与约束

- **安全**：邮箱授权信息需安全存储（如前端不持久化明文密码，使用 OAuth token）；会话若经后端，考虑敏感信息不写日志。  
- **可用性**：主要操作（发邮件、新增待办、发消息）在 3 步内完成；列表加载有明显 loading 状态。  
- **技术约束**：首版可纯前端 + 本地存储（localStorage/IndexedDB）实现日程/待办；邮件需通过 QQ 邮箱开放能力（如 IMAP/SMTP 或官方 API）；会话需后端或 Dify API。  
- **兼容**：现代浏览器（Chrome/Edge/Firefox 等）；如需兼容 IE 需单独说明。

---

## 6. 成功标准 (Success Criteria)

- **SC-01**：用户能在 5 分钟内完成 QQ 邮箱授权并看到收件列表。  
- **SC-02**：用户能成功发送一封邮件并在应用内得到明确成功/失败反馈。  
- **SC-03**：用户能创建、完成、删除待办，并在“今日”视图中看到对应项。  
- **SC-04**：用户能在会话中发送一条消息并收到至少一条助手回复（可为占位或 Dify 返回）。  
- **SC-05**：邮件、日程/待办、会话三个模块在同一工作台内可切换使用，无阻塞性错误。

---

## 7. 实现顺序建议（供你按步实现）

1. **工作台骨架**：布局、导航/Tab（邮件 | 日程与待办 | 会话）。  
2. **日程与待办**：本地存储 + 列表 + 今日视图 + 增删改与完成状态。  
3. **会话**：对话 UI + 对接 Dify（或 mock）返回一条回复。  
4. **邮件**：QQ 邮箱授权（查清使用 IMAP/SMTP 还是开放平台 API）+ 收件列表 + 发件。  
5. **增强**：仪表盘摘要、流式回复、邮件正文查看、多会话等按需迭代。

## 8. 附录：参考与澄清

- **QQ 邮箱**：需确认使用 [QQ 邮箱 IMAP/SMTP](https://service.mail.qq.com/detail/0/75) 还是 [QQ 互联/开放平台](https://connect.qq.com/) 等，二者授权与 API 不同。  
- **Dify**：会话模块可对接 Dify Chat API，需 API Key 与应用 ID。  
- **数据持久化**：日程/待办首版可用 `localStorage` 或 IndexedDB；若需多端同步，后续可加自建后端或第三方服务。
