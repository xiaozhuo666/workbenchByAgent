# 智能工作台助手 — 需求分析文档

**文档类型**：需求分析 (PRD / 需求规格)  
**创建日期**：2025-03-01  
**状态**：Draft  
**目标**：实现一个集成登录、邮件（QQ 邮箱）、日程/待办、会话功能的个人工作台应用，后端采用 Node.js + MySQL，便于后续按模块分步实现。

---

## 1. 项目概述

### 1.1 产品定位

- **名称**：智能工作台助手  
- **定位**：个人效率工作台 Web 应用，集成常用办公能力，可作为简历项目展示对前端、集成与（可选）AI 助手的理解。  
- **核心能力**：  
  - 登录与账号：用户注册/登录，会话保持，工作台功能均需登录后使用  
  - 邮件：使用 QQ 邮箱接收、发送邮件（与当前登录账号绑定）  
  - 日程与待办：创建、查看、管理日程和待办事项  
  - 会话：与智能助手对话（可与 Dify 等后端集成，实现问答、总结、提醒等）
- **技术栈**：前端 React；后端 **Node.js**；数据持久化 **MySQL**（用户、会话、日程/待办、可选会话记录等）。

### 1.2 目标用户与使用场景

- **主要用户**：本人（及可选的面试官演示）。  
- **典型场景**：  
  - 在一个页面内查看今日待办与近期日程，并处理邮件与助手对话。  
  - 通过自然语言让助手帮忙“查今天有什么待办”“总结未读邮件”等（若接入 AI）。

### 1.3 范围与边界

- **在范围内**：  
  - 用户登录/注册与会话保持（后端 Node.js + MySQL 存储用户与会话）。  
  - QQ 邮箱的收件/发件（需授权与 API 或协议支持，授权信息与登录用户绑定并存于后端）。  
  - 日程/待办的增删改查与基本展示（日/周视图可选），数据持久化于 MySQL。  
  - 会话界面与消息展示；可选接入 Dify 等大模型 API 实现智能回复；会话记录可存 MySQL。  
- **暂不在范围**：  
  - 多租户或复杂权限体系（首版单用户/个人使用）。  
  - 企业邮箱、其他邮箱厂商（首版仅 QQ 邮箱）。  
  - 移动端原生 App（以 Web 为主，响应式可选）。

## 2. 用户故事与验收场景

### User Story 0 - 登录与账号 (Priority: P1)

作为用户，我希望能**注册账号并登录**工作台，登录后会话保持，访问邮件、日程、会话等模块时无需重复登录；未登录时只能看到登录/注册页。

**Why this priority**：登录是使用工作台的前提；后端采用 Node.js + Express 提供登录/注册 API，用户与会话信息持久化于 MySQL。

**Independent Test**：注册新账号并登录后，能进入工作台主界面；刷新页面仍保持登录状态；登出后再次访问需重新登录。

**Acceptance Scenarios**:

1. **Given** 未登录，**When** 访问工作台任意功能页，**Then** 重定向至登录页或展示登录/注册入口。  
2. **Given** 在登录页，**When** 输入有效账号与密码并提交，**Then** 校验通过后进入工作台主界面，并下发会话标识（如 Cookie / JWT），后续请求携带该标识。  
3. **Given** 在注册页，**When** 填写用户名、密码（及可选邮箱等）并提交，**Then** 后端（Node.js）校验后写入 MySQL 用户表，并完成自动登录或跳转登录页。  
4. **Given** 已登录，**When** 刷新页面或重新打开应用，**Then** 会话仍有效则保持在工作台，否则引导重新登录。  
5. **Given** 已登录，**When** 点击登出，**Then** 清除会话并返回登录页。

---

### User Story 1 - 日程与待办管理 (Priority: P1)

作为**已登录**用户，我希望能**创建、查看、编辑、完成/删除**日程和待办事项，并在工作台首页或日程页看到今日/本周视图；数据由后端 Node.js 写入 MySQL，与当前用户关联。

**Why this priority**：日程与待办是个人工作台的核心，与邮件、会话并列。

**Independent Test**：能创建一条待办并标记完成，能创建一条带日期的日程并在列表中看到。

**Acceptance Scenarios**:

1. **Given** 在日程/待办模块，**When** 新增一条待办（标题必填，截止时间可选），**Then** 列表中展示该待办，并支持“标记完成”“删除”。  
2. **Given** 在日程/待办模块，**When** 新增一条日程（标题、日期、可选时间），**Then** 在列表或日历视图中展示，并支持编辑、删除。  
3. **Given** 已有若干待办与日程，**When** 打开“今日”或“本周”视图，**Then** 按时间或日期聚合展示，便于快速浏览。  
4. **Given** 某条待办已标记完成，**When** 查看列表，**Then** 可区分已完成与未完成（样式或分区），并可选择隐藏已完成。
### User Story 2 - 邮件收发（QQ 邮箱）(Priority: P1)

作为用户，我希望能用工作台**查看 QQ 邮箱收件列表**并**发送邮件**，这样就不必单独打开 QQ 邮箱网页。使用前须已登录；QQ 邮箱授权与当前登录用户绑定，由后端（Node.js）安全存储。

**Why this priority**：邮件是工作台的核心能力之一，无邮件则“工作台”价值不足。

**Independent Test**：登录后配置 QQ 邮箱授权，能在应用内看到最近 N 封邮件列表，并能成功发送一封测试邮件。

**Acceptance Scenarios**:

1. **Given** 未登录，**When** 尝试打开邮件模块，**Then** 先引导登录后再进入邮件功能。  
2. **Given** 已登录且已配置 QQ 邮箱授权，**When** 打开邮件模块或刷新列表，**Then** 展示收件箱列表（发件人、主题、时间等），支持分页或“加载更多”；拉取与发件由后端 Node.js 代理或转发 QQ 邮箱 API。  
3. **Given** 已登录且在邮件模块，**When** 点击“写邮件”并填写收件人、主题、正文后发送，**Then** 发送成功并给出提示，且可在“已发送”或收件箱中验证（视 API 能力）。  
4. **Given** 已登录但未配置或授权过期，**When** 进入邮件模块，**Then** 展示引导配置/重新授权 QQ 邮箱的入口与说明；授权信息由后端存储于 MySQL（或加密存储），与当前用户关联。

### User Story 3 - 会话功能 (Priority: P2)

作为**已登录**用户，我希望能与**智能助手对话**，发送消息并看到回复；可选地，助手能根据我的日程/待办或邮件摘要回答问题（依赖后端 Node.js 与 Dify 等能力）。

**Why this priority**：会话体现“智能”与集成能力，但首版可先做界面与基础对话，再接 Dify。

**Independent Test**：在会话界面输入一条消息并发送，能收到一条回复（可为固定回复或 Dify 返回）。

**Acceptance Scenarios**:

1. **Given** 在会话模块，**When** 输入文本并发送，**Then** 消息出现在对话列表，并展示助手的回复（流式或一次性均可）。  
2. **Given** 已接入 Dify（或类似 API），**When** 用户问“今天有什么待办？”，**Then** 助手能基于当前用户的待办数据回答（需后端或前端把待办数据传给 API）。  
3. **Given** 网络异常或 API 失败，**When** 用户发送消息，**Then** 界面给出错误提示，且用户可重试。

---

### Edge Cases

- **登录**：用户名重复、密码强度校验、会话过期时间、重复登录与单设备/多设备策略；后端 Node.js 与 MySQL 错误时的友好提示。  
- **邮件**：授权过期、网络超时、无邮件时列表为空、发件失败（如收件人格式错误）时的提示与重试。  
- **日程/待办**：无数据时的空状态；跨天/跨时区显示；大量数据时的分页或虚拟滚动。  
- **会话**：长对话的上下文长度限制；敏感信息不落库或脱敏（若持久化）；未接入 AI 时的占位回复说明。

## 4. 关键实体与数据（概念层）

- **用户 (User)**：属性：用户 ID、用户名、密码哈希、可选邮箱、创建时间等。存于 **MySQL**，由 Node.js 后端读写。  
- **会话 (Session)**：属性：会话 ID、用户 ID、创建/过期时间等。存于 MySQL（或 Redis），用于登录态校验。  
- **邮件 (Email)**：对应一封邮件。属性：发件人、收件人、主题、正文、日期时间、是否已读、邮件 ID 等。来源：QQ 邮箱 API；拉取与发件由 Node.js 代理。  
- **邮箱授权 (MailAccount)**：与用户绑定，属性：用户 ID、QQ 邮箱授权信息（如 token、刷新方式），**由后端 Node.js 安全存储**（MySQL 或加密字段），用于邮件拉取与发送。  
- **待办 (Todo)**：属性：标题、是否完成、截止日期/时间、创建时间、唯一 ID、**用户 ID**。存于 **MySQL**，由 Node.js 提供 CRUD API。  
- **日程 (Event/Schedule)**：属性：标题、日期、开始/结束时间、备注、唯一 ID、**用户 ID**。存于 **MySQL**，由 Node.js 提供 CRUD API。  
- **会话消息 (ChatMessage)**：属性：角色（user/assistant）、内容、时间戳、可选会话 ID、**用户 ID**。用于展示；可选存 MySQL 供历史查询。

## 5. 非功能需求与约束

- **安全**：用户密码须在后端加密存储（如 bcrypt）；邮箱授权信息仅存于后端（Node.js + MySQL 或加密字段），前端不持久化明文密码或 token；会话若经后端，敏感信息不写日志。  
- **可用性**：登录/注册流程简洁；主要操作（发邮件、新增待办、发消息）在 3 步内完成；列表加载有明显 loading 状态。  
- **技术约束**：**后端采用 Node.js**（如 Express/Koa），**持久化采用 MySQL**（用户、会话、日程/待办、邮箱授权等）；前端 React 通过 REST/HTTP 调用后端 API。邮件需通过 QQ 邮箱开放能力（如 IMAP/SMTP 或官方 API），由后端代理或转发；AI 能力优先接入阿里云百炼（OpenAI 兼容模式，可使用 `openai` SDK + `qwen-plus`）。  
- **兼容**：现代浏览器（Chrome/Edge/Firefox 等）；如需兼容 IE 需单独说明。

## 6. 成功标准 (Success Criteria)

- **SC-00**：用户能完成注册并登录，刷新页面后仍保持登录状态；登出后需重新登录才能进入工作台。  
- **SC-01**：已登录用户能在 5 分钟内完成 QQ 邮箱授权并看到收件列表。  
- **SC-02**：用户能成功发送一封邮件并在应用内得到明确成功/失败反馈。  
- **SC-03**：用户能创建、完成、删除待办（数据存 MySQL），并在“今日”视图中看到对应项。  
- **SC-04**：用户能在会话中发送一条消息并收到至少一条助手回复；AI 回复优先使用 SSE 流式返回，以提升感知速度与交互体验。  
- **SC-05**：登录后，邮件、日程/待办、会话三个模块在同一工作台内可切换使用，无阻塞性错误。

## 7. 实现顺序建议（供你按步实现）

1. **后端基础与登录（Node.js + MySQL）**：初始化 Node.js 项目与 MySQL 库表（用户、会话）；实现注册、登录、登出 API；会话保持（JWT 或 Session）。  
2. **首页日程与待办 + 百炼 AI**：先在首页实现日程/待办管理基础能力（新增、展示、完成），并接入阿里云百炼（OpenAI 兼容模式）实现“AI 生成待办列表”与“AI 完成待办项”能力。  
3. **AI 体验优化（SSE 流式传输）**：对 AI 相关接口改造为 SSE 流式输出；前端支持逐段渲染、加载态与中断/重试，重点优化“AI 生成待办列表”“AI 完成待办项”两类交互体验。  
4. **会话**：对话 UI + 后端对接 Dify（或 mock）返回一条回复；可选会话记录存 MySQL。  
5. **邮件**：QQ 邮箱授权（与当前用户绑定，授权信息存后端）+ 收件列表 + 发件（由 Node.js 代理/转发）。  
6. **增强**：仪表盘摘要、流式回复、邮件正文查看、多会话等按需迭代。

## 8. 附录：参考与澄清

- **Node.js**：推荐 Express 或 Koa；需提供 REST API（登录、注册、登出、日程/待办 CRUD、邮件代理、会话转发等）；环境变量管理敏感配置（数据库、JWT 密钥、QQ 邮箱/Dify 等）。  
- **MySQL**：至少包含用户表、会话表、待办表、日程表；邮箱授权可单独表或加密字段与用户关联；可选会话消息表。  
- **QQ 邮箱**：需确认使用 [QQ 邮箱 IMAP/SMTP](https://service.mail.qq.com/detail/0/75) 还是 [QQ 互联/开放平台](https://connect.qq.com/) 等，二者授权与 API 不同；由 Node.js 后端调用，授权信息存 MySQL。  
- **阿里云百炼（OpenAI 兼容模式）**：待办 AI 能力通过 `openai` SDK 调用，`baseURL` 使用 `https://dashscope.aliyuncs.com/compatible-mode/v1`，模型可用 `qwen-plus`；API Key 使用环境变量 `DASHSCOPE_API_KEY`。  
- **示例（用于后端服务）**：
  ```js
  import OpenAI from "openai";

  const openai = new OpenAI({
    apiKey: process.env.DASHSCOPE_API_KEY,
    baseURL: "https://dashscope.aliyuncs.com/compatible-mode/v1"
  });

  async function main() {
    const completion = await openai.chat.completions.create({
      model: "qwen-plus",
      messages: [
        { role: "system", content: "You are a helpful assistant." },
        { role: "user", content: "请根据今天安排生成待办列表，并可按指令完成待办。" }
      ]
    });
    console.log(JSON.stringify(completion));
  }

  main();
  ```
