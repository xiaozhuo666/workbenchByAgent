## 5.票务出行页面实现

### 5.1 需求规格（Spec）

#### 5.1.1 产品目标与范围

目标：将“聊天查票”升级为“任务接力”，形成 AI 决策 + 结构化票务工作台的连续体验，而非仅 query 跳转填表。

范围（MVP）：
1. 新增 `票务出行` 页面（搜索表单 + 结果列表 + AI 建议区）；
2. 聊天识别查票意图后返回“行程卡”；
3. 点击“查看结果”后跳转 `/tickets?draftId=xxx`，页面自动填充并自动触发查询；
4. 结果区给出 AI 三方案（最快/最便宜/最舒适）及理由；
5. 预留“直达/中转优选”双 Tab，MVP 可先保证直达闭环。

不在 MVP 范围：
1. 自动下单、自动支付、自动改签；
2. 复杂多乘客规则引擎；
3. 多城市联程一次性求解。

---

#### 5.1.2 信息架构（IA）

页面：`/tickets`（票务工作台）

1. 顶部搜索区（固定）
   - 字段：出发地、到达地、出发日期、车次类型（高铁/动车/全部）；
   - 操作：交换出发到达、查询、重置、来自 AI 标签（含 draftId）。
2. 主体结果区（中部）
   - Tab：`直达`、`中转优选`；
   - 结果列表：车次、发到时间、历时、票价、余票状态；
   - 工具栏：排序（最早出发/最短耗时/最低价格）、筛选（二等座可售、发车时段等）。
3. AI 建议区（右侧桌面 / 抽屉移动端）
   - 推荐卡：最快、最便宜、最舒适；
   - 每张卡给“为什么推荐”（耗时/价格/换乘风险/余票稳定性）；
   - 操作：一键套用到筛选条件、重新生成建议。

交互入口：聊天助手中的“行程卡”
1. 展示解析结果（日期、出发地、到达地、偏好）；
2. 按钮：`查看结果`（跳转+接力）、`继续细化`（补参数）。

#### 5.1.3 结构化对象与字段定义

`tripDraft`（聊天层 -> 票务页接力核心对象）：

```json
{
  "draftId": "td_20260305_xxx",
  "source": "ai_assistant",
  "route": {
    "fromCity": "上海",
    "fromStationCode": "SHH",
    "toCity": "北京",
    "toStationCode": "BJP"
  },
  "date": "2026-03-06",
  "preferences": {
    "trainTypes": ["G", "D"],
    "departureTimeRange": "06:00-12:00",
    "seatTypes": ["二等座"],
    "strategy": "fastest"
  },
  "passengerProfile": {
    "count": 1,
    "priority": "准点优先"
  },
  "status": "ready",
  "createdAt": "2026-03-05T10:00:00Z",
  "expiresAt": "2026-03-05T12:00:00Z"
}
```

字段约束：
1. `draftId` 为唯一键，页面仅通过 `draftId` 拉取，不通过长 query 传输业务参数；
2. `status`：`collecting` / `ready` / `expired`；
3. `expiresAt` 超时后需提示用户一键重查；
4. 站点优先用站码（与 12306 工具对齐），城市名用于展示。

#### 5.1.4 前后端联动协议（API Contract）

1. 创建草稿（聊天侧）
   - `POST /api/ticket-drafts`
   - 入参：意图解析结果（route/date/preferences/passengerProfile）
   - 出参：`draftId`、`status`、`expiresAt`

2. 获取草稿（票务页加载）
   - `GET /api/ticket-drafts/:draftId`
   - 出参：完整 `tripDraft`

3. 基于草稿查询车票
   - `POST /api/tickets/search`
   - 入参：`draftId` + 可选覆盖项（排序、筛选）
   - 出参：`directOptions[]`、`transferOptions[]`、`meta`（耗时、命中条数、数据时间）

4. AI 三方案推荐
   - `POST /api/tickets/recommendations`
   - 入参：`draftId` + 结果集摘要（或服务端内部复用查询结果）
   - 出参：`fastest` / `cheapest` / `comfortable`，每项含 `reason[]` 与 `confidence`

错误码建议：
1. `DRAFT_NOT_FOUND`：草稿不存在；
2. `DRAFT_EXPIRED`：草稿过期；
3. `MCP_UPSTREAM_TIMEOUT`：12306 工具超时；
4. `MCP_UPSTREAM_PARTIAL`：部分可用，需前端展示“结果可能不完整”。

#### 5.1.5 AI 与票务层职责分工

1. AI 聊天层
   - 查票意图识别；
   - 参数补全与追问；
   - 生成行程卡与 `tripDraft`；
   - 推荐解释文案生成。

2. 票务页面层
   - 稳定查询、筛选、排序；
   - 结果可视化对比；
   - 推荐卡展示与交互。

3. MCP 工具层（12306）
   - 站码映射；
   - 余票查询；
   - 过站查询；
   - 中转查询。

#### 5.1.6 UI/UX 设计约束（ui-ux-pro-max）

基线规则（用于开发过程验收）：
1. 页面风格采用“专业、可信、可读性优先”，避免花哨动画干扰信息比对；
2. 所有交互元素提供清晰 hover/focus 态，过渡建议 150-300ms；
3. 颜色对比满足可读性（正文至少 4.5:1）；
4. 移动端使用抽屉承载 AI 建议，桌面端使用右侧面板；
5. 禁止使用 emoji 作为功能图标，统一 SVG 图标集；
6. 关键断点至少覆盖 375/768/1024/1440。

### 5.2 技术实现计划（Plan）

#### Phase A：接力链路打通（MVP 主线）

目标：
1. 打通“聊天识别 -> 行程卡 -> draftId 跳转 -> 票务页自动查询”闭环；

产出物：
1. `ticket_drafts` 数据表（或复用现有会话表扩展）；

3. `/tickets` 页面基础布局与自动填充逻辑；
4. 聊天消息中的行程卡组件（含“查看结果/继续细化”）。

风险与规避：

2. 规避：后端入库前做 schema 校验，`status=ready` 才允许跳转查询。

DoD：
1. 从聊天点击“查看结果”后 2 步内看到结果列表；
2. 刷新页面可通过 `draftId` 重建状态；
3. 草稿过期有明确提示和“重新生成”入口。

#### Phase B：决策增强与中转能力

目标：
1. 加入 AI 三方案推荐；
2. 增加“中转优选”Tab 与策略（最短总时长/最少等待）。

产出物：
1. `tickets/recommendations` API；
2. 推荐卡 UI 与解释文案模板；
3. 中转结果模型与排序策略。

风险与规避：
1. 风险：推荐与用户筛选冲突，造成“看不懂为什么推荐”；
2. 规避：推荐卡明确显示“推荐依据字段”，并支持“一键套用筛选”。

DoD：
1. 三方案均可追溯到具体车次方案；
2. 推荐理由至少包含 2 个可量化指标；
3. 中转方案可按总历时、等待时长稳定排序。

#### Phase C：稳定性、灰度与验收

目标：
1. 完成异常降级、观测指标与灰度开关；

DoD：

3. 回归用例通过率达成团队基线。
