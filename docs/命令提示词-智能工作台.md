# specify 命令提示词（第一个功能：登录/注册）

请为「智能工作台助手」创建第一个功能规格：实现基本的登录/注册能力。
1.范围仅包含注册、登录、会话保持、登出与未登录拦截，不包含忘记密码、邮箱验证、第三方登录。
系统约束：
1.系统约束：后端使用 Node.js 提供接口，前端使用 React 调用接口，
2.前端样式与组件采用 Less + Ant Design，数据库使用 MySQL 存储用户与会话数据。请严格对齐 `docs/智能工作台助手-需求分析.md` 中 User Story 0（登录与账号）、功能需求 FR-A01 至 FR-A05、关键实体 User/Session、Edge Cases（用户名重复、密码强度校验、会话过期、后端或数据库错误提示）与成功标准 SC-00。验收场景至少覆盖：1）未登录访问工作台功能时跳转登录页；2）登录成功后进入工作台并获得会话标识；3）注册时校验唯一性并写入用户表；4）刷新页面保持登录；5）登出后清除会话并返回登录页。请输出清晰、可测试、可直接进入 plan 阶段的 spec。


# plan部分---技术相关的文档

请基于 `specs/001-auth-login-register/spec.md` 生成本功能的技术实现计划（plan），目标是落地“登录/注册”MVP。请输出可执行的技术方案，覆盖架构、数据模型、接口、任务拆分与风险控制，且与 `docs/智能工作台助手-需求分析.md` 保持一致。

技术约束（必须遵守）：
1. 后端：Node.js + Express。
2. 前端：React。
3. UI：Less + Ant Design。
4. 数据库：MySQL。
5. 前端仅调用后端 API，不直接连接数据库。
6. 鉴权方案固定：JWT（不再做 Session 二选一）。
7. QQ 邮箱连接方案固定：SMTP/IMAP；授权码通过环境变量注入（例如 `QQ_MAIL_AUTH_CODE`），不得写入代码或文档明文。

计划内容至少包含：
1. 架构与目录：前后端模块划分、建议目录结构、环境变量清单（如 DB、JWT、端口）。
2. 数据模型：`users`、`sessions`（或等价会话方案）表结构草案，字段、索引、唯一约束（如 username/email）。
3. API 设计：注册、登录、登出、获取当前用户信息（用于登录态恢复）四类接口；请求/响应示例；错误码与错误信息规范。
4. 鉴权方案：使用 JWT；说明 access token 存放方式、过期策略、刷新页面保持登录、未登录拦截的实现路径。
5. 安全要求：密码哈希（bcrypt）、基础输入校验、敏感信息不落日志、最小权限原则。
6. 前端流程：登录页/注册页表单校验、提交与错误提示、登录成功跳转、路由守卫、登出流程。
7. 测试计划：最小测试集合（注册成功、重复用户名、登录失败、登录态恢复、登出失效）。
8. 任务拆分：按“后端先行 -> 前端接入 -> 联调 -> 测试”给出可执行 tasks，并标注优先级和依赖关系。
9. 风险与决策：列出关键风险（如会话策略、跨域、QQ 邮箱后续集成兼容性）及规避方案。

范围边界（本阶段不做）：
1. 忘记密码、邮箱验证、第三方登录。
2. 多租户与复杂权限系统。
3. 邮件/日程/会话业务功能实现（本 plan 只做登录注册基础能力）。
