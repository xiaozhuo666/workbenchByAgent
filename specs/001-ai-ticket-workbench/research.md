# Phase 0 Research - AI 票务工作台接力体验

## 决策 1：聊天到页面采用 `draftId` 接力

- **Decision**: 聊天层仅传递 `draftId` 到票务页，业务字段由票务页通过草稿接口获取。  
- **Rationale**:
  - 避免 URL 携带大量业务参数导致链路脆弱；
  - 便于后续扩展更多偏好字段而不破坏跳转协议；
  - 支持草稿状态管理（collecting/ready/expired）与审计追踪。  
- **Alternatives considered**:
  - 直接 query 传完整参数：实现快，但字段扩展与安全性较差；
  - 本地缓存传递：跨设备与刷新可恢复性不足。

## 决策 2：草稿生命周期采用“有效期 + 可恢复”

- **Decision**: 行程草稿设置有效期，过期后必须给出“重新生成/手动调整后重查”路径。  
- **Rationale**:
  - 票务数据时效性强，过期草稿直接复用风险高；
  - 显式恢复路径可避免用户陷入无结果状态；
  - 与“连续体验”目标一致，减少用户重复输入。  
- **Alternatives considered**:
  - 永不过期：可能导致旧参数误用；
  - 过期直接报错并阻断：用户体验差、流程中断明显。

## 决策 3：推荐卡必须“可解释 + 可执行”

- **Decision**: 三方案推荐（最快/最便宜/最舒适）均包含理由，并支持一键映射为筛选条件。  
- **Rationale**:
  - 仅给结论不利于建立用户信任；
  - 推荐与筛选联动可直接转化为实际操作，形成闭环；
  - 降低“看懂但做不到”的落差。  
- **Alternatives considered**:
  - 仅展示推荐文案：缺少可执行性；
  - 只提供排序不做推荐：差异化不足。

## 决策 4：查询与推荐分层降级

- **Decision**: 查询结果与推荐结果独立处理，推荐失败不影响基础列表；上游异常时给出可恢复提示。  
- **Rationale**:
  - 保障核心“可查可筛”能力优先可用；
  - 降低单点失败对主流程的影响；
  - 更符合生产可用性和灰度发布策略。  
- **Alternatives considered**:
  - 强依赖推荐接口：用户在推荐异常时无法继续操作；
  - 失败静默：用户无法判断结果完整性，决策风险高。

## 决策 5：界面布局采用“工作台”而非单页搜索

- **Decision**: 页面采用顶部搜索 + 中部结果 + 右侧（移动端抽屉）AI 建议面板。  
- **Rationale**:
  - 支持连续比对与快速调整；
  - 便于承载推荐解释与一键套用；
  - 满足 UI 规范中响应式、可读性、交互反馈要求。  
- **Alternatives considered**:
  - 传统单列表搜索页：承载 AI 决策信息不足；
  - 完全对话式结果展示：不利于稳定筛选和对比。

## 结论

所有 Technical Context 中潜在不确定项已完成决策，无 NEEDS CLARIFICATION 残留，可进入 Phase 1 设计与契约定义。
